# 五套主题语义校准与共同 UI 微调设计

日期：2026-07-03
状态：设计已批准，待实现计划

## 背景

AI HOT Notifier 是一个 420px × 600px 的 Manifest V3 浏览器扩展弹窗。它的核心界面由顶部操作区、可折叠设置面板和热点列表组成。当前已有 5 套主题：

- 墨夜（`dark`）
- 暗森（`green-dark`）
- 铬墨（`chrome-dark`）
- 晨雾（`clear-light`）
- 石青（`slate-night`）

用户希望从第一性原理出发，对当前 5 套主题做全方位对比，并在不推翻现有主题体系的前提下，尽可能提升用户观感和使用体验。

## 目标

- 保留现有 5 套主题及其设置入口，不删除主题，不改变用户已保存的 theme key。
- 明确每套主题的视觉任务和使用场景，避免只是换色。
- 优化列表阅读体验，让用户在数秒内判断未读、已读、特关、分类和时间信息。
- 微调共同 UI 细节，包括 hover、未读强调、标签、日期浮标、设置面板、按钮和表单控件状态。
- 继续保持扩展轻量、无构建依赖、不新增 npm/package 流程。
- 遵守现有弹窗约束：固定尺寸、设置面板默认只展开常规、特关规则首行布局、非输入区域不出现文本插入光标。

## 非目标

- 不重新设计信息架构，不新增独立设置页。
- 不引入外部字体、图片资源或构建工具。
- 不修改轮询、通知、存储、特关匹配等业务逻辑。
- 不改变主题选择项的 storage key 或已有配置兼容性。
- 不做大幅动效或卡片浮起阴影，避免在小弹窗内产生噪声。

## 第一性原理

弹窗的首要任务不是品牌展示，而是热点扫描：

> 用户在 3 秒内判断：有没有重要新热点？哪些已读？哪些值得点开？

因此设计优先级为：

1. 信息层级清楚：未读、特关、分类、时间、摘要各有独立权重。
2. 长时间低负担：深色主题不能依赖高饱和色硬撑，浅色主题不能刺眼。
3. 主题各自有场景：默认、夜读、系统工具、日间阅读、工程高密度扫描。
4. 状态语义一致：hover、focus、loading、成功、失败在所有主题中表达同一含义。

## 主题定位

| 主题 | 当前问题 | 优化定位 | 重点调整 |
| --- | --- | --- | --- |
| 墨夜 | 默认黑底稳定，但层次略平，橙色容易被泛用 | 默认主题，强调“热点雷达” | 保持深黑，提高未读层次，降低橙色在普通 hover 中的使用频率 |
| 暗森 | 低刺激，但文字、摘要、元信息梯度不够明确 | 夜间长读、护眼 | 降低绿色饱和度，拉开正文/摘要/元信息对比 |
| 铬墨 | 系统感好，但和墨夜差异不足 | 浏览器原生、干净工具感 | 更冷、更轻，按钮和边框靠近 Chrome/Edge 工具 UI |
| 晨雾 | 浅色清爽，但层级偏灰，未读不够明确 | 日间阅读、轻量新闻流 | 使用非纯白背景和浅蓝灰未读层，不刺眼 |
| 石青 | GitHub 感明确，但边框偏重，噪声略大 | 工程师高密度阅读 | 保留代码平台气质，降低边框噪声，提高 hover 辨识 |

## 共同 UI 设计

### 列表结构

列表继续沿用现有布局，不改 DOM 主结构：

```text
┌ header: logo                         actions ┐
├ settings panel, collapsed by default          ┤
│                                                │
│ unread item  ┃ title                           │
│              ┃ summary                         │
│              ┃ tag · source · time             │
│                                                │
│ read item      title muted                     │
│                summary muted                   │
└ sticky date badge                              ┘
```

### Hot rail

本次设计的标志性元素是 **Hot rail**：未读/特关条目左侧的主题色热度轨道。

规则：

- 未读条目使用 2px 左侧主题色轨道和轻背景。
- 特关条目沿用同一语义，但轨道颜色可更接近 accent，避免额外大色块。
- 已读条目不显示强轨道，保持安静。
- 轨道不增加布局宽度，不影响标题和摘要可读区域。

设计理由：

- 比大面积背景高亮更节省空间。
- 比纯标签更容易扫视。
- 符合 AI HOT “热点雷达/读数”的产品语义。

### 未读与已读

- 未读标题使用略高字重，不依赖高饱和色。
- 未读摘要和 meta 保持清楚，但低于标题。
- 已读标题、摘要、meta 同步降权。
- 已读 hover 后恢复到可辨认颜色，方便二次查找。

### 分类标签与特关标签

- 分类标签保持小尺寸，不增加圆角和阴影。
- 背景透明度降低，避免和标题竞争。
- 特关标签继续保留，颜色使用主题 accent，背景使用 `--accent-soft`。
- 标签是辅助信息，不应成为列表最醒目的元素。

### 日期浮标

- 日期浮标继续 sticky。
- 背景使用 surface，而不是强对比底色。
- 边框使用 light border，整体降低按钮感。
- 文本保持 meta 级别。

### 设置面板

- 设置面板仍默认只展开“常规”。
- 分组标题更像可展开导航：小字、足够点击高度、打开时使用 accent。
- select/button/input hover 不全部变成 accent 大面积反馈，只在文字或边框轻微响应。
- switch 继续使用 accent，因为它表示明确开关状态。
- 特关规则卡片保留左侧 accent 线，但减少灰盒堆叠感。
- 特关规则首行继续保持“来源 / 作者 / 停用 / 删除”同行；关键词仅存在时另起一行。

### 动效

- 保留 logo 红点呼吸、刷新 loading、按钮确认反馈。
- hover 仅做轻微背景/颜色变化。
- 不新增滚动动画、浮起阴影或复杂过渡。
- 继续尊重 `prefers-reduced-motion`。

## 实现边界

主要改动集中在：

- `popup.html`：主题 CSS token、公共 UI 样式、列表/设置面板/标签/日期浮标状态。
- `popup.js`：如主题背景映射需要同步，则仅更新 `applyTheme()` 中的背景色和 color-scheme 判断。
- `test-popup-ui.js`：如已有静态约束对颜色或样式有断言，需要同步更新。
- `README.md`：如当前文档仍写“三套深色主题”，应更新为五套主题描述。
- `manifest.json`：实现完成后按仓库发布流程升级 patch 版本。

不应改动：

- `background.js` 的业务逻辑。
- 特关匹配、通知节奏、存储结构。
- host permissions。
- 打包流程和依赖管理方式。


## 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 主题 token 改动导致部分状态对比不足 | 逐套检查正文、摘要、meta、已读、未读、hover、标签 |
| 浅色主题未读背景过重 | 晨雾使用浅蓝灰，不使用高饱和蓝 |
| 深色主题之间差异仍不足 | 墨夜偏热、暗森偏木质、铬墨偏冷工具、石青偏 GitHub 工程 |
| 共同 UI 调整破坏特关规则布局 | 不改 DOM 主结构，只调整局部 CSS，并运行 `test-popup-ui.js` |
| hover 反馈过弱 | 保留边框/背景/文字三者之一响应，但避免所有控件都抢 accent |

## 自检结果

- 无占位符或待定项。
- 设计范围聚焦在主题 token 和共同 UI 微调，不包含业务逻辑重构。
- 主题名称、storage key、弹窗尺寸、设置面板分组和特关布局约束均保持兼容。
- 实现文件、测试命令、风险边界明确。
